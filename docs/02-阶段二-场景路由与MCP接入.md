# 02 · 阶段二：场景路由 + MCP 接入

> **本阶段目标**：让每个问题场景都有"专门的对方"，且已连通验证。
> **完成标志**：你抛出的每类问题都能路由到对应数据源/技能/连接器，且已发探针验证连通。

---

## 2.1 为什么需要路由表

董办问题跨度极大：问工商穿透、问法规条文、问公告案例、问实时行情——用单一源硬扛必然串台或编造。路由表就是"派单机制"：先判场景，再派给那一行专门的「对方」。

---

## 2.2 建场景路由表（用模板）

打开 `references/scenario-routing-template.md`，按三步建表：

1. **枚举高频场景**：把你公司董办最常问的 8–12 类问题列出来（制度/法规/判例/公告/工商/董监高/委员会/行情/历史届次…）。
2. **指派专用源**：每类问题指定**唯一**主数据源/技能/连接器（见下方"常用源清单"）。
3. **写说明 + 定兜底**：每行注明"为什么去这"；再写一条兜底顺序（专用源不可用时的补位链）。

**常用源清单（按场景）**：

| 场景 | 专用源 |
|---|---|
| 制度/内部权限 | 自建知识库（IMA 等） |
| 法规/司法解释 | 北大法宝 PKULaw |
| 判例/裁判要点 | 北大法宝 PKULaw |
| 公告/监管案例 | announcement-case-analyzer 技能（cninfo） |
| 工商/股权穿透 | 企查查 qcc / 天眼查 tyc |
| 现任董监高 | 通达信 F9（tdx-connector） |
| 委员会构成 | 通达信 F9 + 自建知识库 |
| 最新公告/函件 | 交易所/巨潮/公司官网 |
| 行情/新闻/检索 | Tavily |
| 实时行情/估值 | 通达信（tdx-connector） |
| 历史届次董事会 | cninfo 换届公告 + 新浪财经（联网交叉） |

> ⚠️ 公告案例输出须统一遵循 announcement-case-analyzer 技能内已定义的 **19 类事件枚举 + 系列公告合并规则**，本体不再重复定义，避免双源不一致。

---

## 2.3 源能力矩阵（单点发现 → 全域覆盖）

每个源一旦在某场景验证可用，必须把它**全部数据模块**摊开、挂到所有需要它的场景——别只在发现它的那一行补一列。详见 [04 · 阶段四：经验泛化与持续优化](./04-阶段四-经验泛化与持续优化.md)。

---

## 2.4 MCP 接入实战

逐个连接器接入，填 `references/mcp-wiring-checklist.md` 的清单。每个连接器记录：用途 / 启用方式 / 数据模块 / 兜底 / 验证命令。

常用接入项：

- **企查查 qcc-company**：股权穿透、董监高、对外投资。启用：在 WorkBuddy 连接器页授权。
- **天眼查 tyc-mcp**：160+ 企业数据能力，企查查的补充/兜底。
- **通达信 tdx-connector**：F9 深度资料、行情、财务。
- **北大法宝 pklaw**：法规/判例检索（注意：重启 WorkBuddy 后可能需刷新授权）。
- **IMA（ima-mcp / ima-skill）**：知识库检索与回写。
- **announcement-case-analyzer**：公告案例检索（cninfo API）。
- **Tavily（tavily-mcp）**：联网深度检索兜底。

> 实战坑：北大法宝等连接器重启后授权可能掉，表现为 `disconnected`；此时当场告知用户"法规源暂不可用，请重启 WorkBuddy 刷新授权"，**不得绕道其他源编造法规**。

---

## 2.5 连通验证（必做）

每个源接上后，发一个**探针问题**验证它能回、回得对：

- 企查查：查一家你熟悉的子公司股东 → 看能否返回穿透。
- 通达信：查目标公司董监高 → 看名单/任职日期是否对。
- 北大法宝：搜一条你熟知的法条 → 看是否返回原文链接。
- Tavily：搜一条近期行业新闻 → 看是否返回结构化结果。

验证不过的源，先在路由表标注"⚠️ 未连通"，不要假装能用。

---

## 2.6 自检清单

- [ ] 路由表覆盖你公司高频场景（≥8 类）
- [ ] 每类场景有唯一专用源 + 兜底顺序
- [ ] 所有计划接入的 MCP 已连通并探针验证
- [ ] 未连通的源已在路由表标注，未编造
- [ ] 公告案例已指向 announcement-case-analyzer，本体不重复定义
